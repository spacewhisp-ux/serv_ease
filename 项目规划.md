# 客服应用项目规划

## 1. 项目目标

目标是做一个以 iOS App Store 上架为优先的客服类应用，先实现一个审核风险低、功能闭环完整、能真实服务用户的 MVP，再逐步扩展实时聊天、AI 助手和后台运营能力。

这个项目当前仍是 Flutter 初始脚手架，应用逻辑基本只在 `lib/main.dart`，尚不具备客服产品功能，因此本规划同时包含：
- 当前项目可上架性审计
- 结合 Apple 审核指南的产品约束
- 多个可选产品方案
- 推荐 MVP 路线
- 上架前清单与实施节奏

## 2. 当前项目审计

### 当前状态
- 目前是近乎原生 Flutter 模板工程
- 尚无客服业务页面、登录、消息、FAQ、工单、设置、隐私页等产品能力
- 尚无后端对接、推送、数据存储、用户身份体系
- 设计上有 `DESIGN.md` 可作为后续 UI 风格基线

### 审计结论
当前项目以“客服应用”身份提交审核时，**不能上架**，核心原因如下：
- 没有明确且可验证的用户价值，容易触发 4.2 最低功能要求风险
- 没有完成的业务闭环，容易触发 2.1 App Completeness 风险
- 没有隐私说明、账户删除、客服联系方式、内容治理机制
- 若直接加聊天但没有举报/屏蔽/审核规则，容易触发 1.2 User-Generated Content 风险

结论：第一版必须做成一个**有真实客服闭环的品牌型服务应用**，而不是泛社交聊天工具。

## 3. 与审核最相关的约束

下面这些是客服类 App 最容易踩中的审核点。

### 3.1 推荐的产品定位
最容易过审的定位是：

**“某个具体业务/品牌的客户支持应用”**，例如：
- 电商售后支持
- SaaS 产品工单与客服
- 本地生活服务售后支持
- 设备/硬件品牌用户支持

这类应用的核心是帮助用户解决真实问题，而不是让陌生人互相聊天。

### 3.2 重点审核风险

#### A. 2.1 App Completeness
必须提供可实际使用的完整体验：
- 登录流程可用
- 核心页面不是占位页
- 客服提交、查看、回复流程闭环可跑通
- 链接、按钮、上传、通知、退出登录都有效

#### B. 4.2 Minimum Functionality
不要提交“只是官网壳子”或“只有一个网页客服入口”的应用。需要有 App 内独立价值，例如：
- 工单状态追踪
- 订单/服务单查询
- 消息通知
- FAQ 搜索
- 图片上传举证
- 售后记录留存

#### C. 1.2 User-Generated Content
只要有聊天、留言、上传图片、工单描述，就已经接近 UGC 范畴，建议首版就具备：
- 举报入口
- 拉黑/屏蔽入口（至少能屏蔽客服外的陌生联系，如果不存在用户互聊，可简化）
- 敏感内容处理规则
- 客服联系方式或申诉联系方式
- 内容审核与删除机制

如果是“随机聊天”“匿名匹配聊天”“陌生人陪聊”方向，2026 年后审核风险明显更高，不适合作为首发。

#### D. 5.1 Privacy
必须明确：
- 收集哪些数据：手机号、邮箱、订单号、聊天记录、图片、设备信息
- 为什么收集：客服处理、订单核验、消息通知
- 是否与第三方共享
- 如何删除账号和数据

如果 App 允许注册账号，建议首版就把**账号注销**放进设置页，避免后续返工。

#### E. 2.3 Accurate Metadata
App Store 标题、截图、描述、隐私标签必须和 App 真实能力一致：
- 不要宣传“7x24 AI 智能客服”，除非真的有
- 不要展示未上线功能截图
- 不要写“秒回”“实时人工”这类不可保证承诺

#### F. 3.1.1 In-App Purchase
如果未来卖的是数字会员、数字增值支持、AI 配额等，需要评估 IAP。
如果只是围绕线下/实物/既有服务的售后支持，通常不必为了“联系客服”本身接 IAP。

#### G. 登录相关
- 如果只支持自有手机号/邮箱登录，问题不大
- 如果提供 Google、Facebook 等第三方登录，通常还要考虑 Sign in with Apple

## 4. 可选产品方案

下面给出 4 个适合首版评估的方案，按“审核友好度”和“上线难度”综合排序。

### 方案 A：FAQ + 工单 + 联系客服（推荐）

**定位**
- 面向某个自有业务的客户支持 App
- 用户先自助检索，再发起工单，必要时联系人工

**核心功能**
- 登录/注册
- FAQ 分类与搜索
- 创建工单
- 上传图片凭证
- 查看处理进度
- 客服回复通知
- 设置页：隐私政策、用户协议、账号注销、联系我们

**优点**
- 审核风险最低
- MVP 闭环最清晰
- 没有“陌生人聊天”风险
- 可以先不用实时 IM，开发成本可控

**缺点**
- 即时沟通感弱
- 用户体验相对传统

**适合你如果**
- 你想尽快上架
- 还没有成熟客服后台
- 希望先验证需求再扩实时聊天

**审核判断**
- 最像“真实服务工具”，最适合首发

### 方案 B：1 对 1 在线客服聊天

**定位**
- 用户与品牌客服单聊
- 不允许用户之间互聊

**核心功能**
- 登录
- 会话列表
- 在线客服聊天
- 常见问题快捷入口
- 图片发送
- 消息已读状态
- 离线时转工单
- 设置与隐私页

**优点**
- 用户感知最像真正客服应用
- 支持效率高
- 可以结合推送形成日常触达

**缺点**
- 审核要求更高
- 需要处理聊天内容治理
- 需要 WebSocket/推送/客服工作台支持

**适合你如果**
- 你已经有客服人员或后台系统
- 你能保证在线响应体验

**审核判断**
- 可上架，但要明确它是“品牌客服聊天”，不是开放聊天社区

### 方案 C：AI 客服助手 + 人工转接

**定位**
- 先由 AI 回答常见问题，复杂问题再转人工

**核心功能**
- FAQ 知识库
- AI 问答
- 人工转接
- 历史会话
- 敏感问题兜底提示
- 反馈“回答是否有帮助”

**优点**
- 运营成本低
- 扩展性强
- 用户首响快

**缺点**
- 容易因宣传与实际不符触发 2.3 风险
- 需要额外处理 AI 幻觉、敏感回复、错误建议
- 对客服质量治理要求更高

**适合你如果**
- 你已有稳定知识库
- 愿意投入 AI 评测和人工兜底

**审核判断**
- 能上，但不建议作为第一版唯一核心能力

### 方案 D：订单/服务单驱动的售后助手

**定位**
- 围绕订单、维修单、预约单展开客服
- 用户不是直接“找人聊天”，而是围绕一个服务对象发起支持

**核心功能**
- 登录
- 订单/服务单列表
- 某单据详情
- 发起售后
- 上传图片/视频
- 进度追踪
- 与客服沟通

**优点**
- 业务场景非常真实
- 审核解释成本低
- 更容易体现 App 独立价值

**缺点**
- 需要已有业务数据源
- 首版后端对接稍重

**适合你如果**
- 你本身就有订单、预约、维修、物流等实体业务

**审核判断**
- 如果业务基础存在，这是非常好的上架方向

## 5. 方案对比

| 方案 | 上架友好度 | 开发复杂度 | 运营复杂度 | 用户体验 | 推荐级别 |
|------|------------|------------|------------|----------|----------|
| A FAQ+工单 | 很高 | 低 | 低 | 中 | 很高 |
| B 1对1聊天 | 高 | 中高 | 中高 | 高 | 高 |
| C AI+人工 | 中 | 中高 | 高 | 中高 | 中 |
| D 订单驱动售后 | 很高 | 中高 | 中 | 高 | 很高 |

## 6. 推荐路线

### 推荐 1：如果你想最快上架
选 **方案 A：FAQ + 工单 + 联系客服**。

这是最稳的 MVP：
- 有真实业务价值
- 不依赖复杂实时系统
- 审核好解释
- 后续可以平滑升级到实时聊天

### 推荐 2：如果你已有成熟业务后台
选 **方案 D：订单/服务单驱动的售后助手**。

这是最“像一个完整商业应用”的方案，天然比泛客服更容易说服审核。

### 不推荐作为首发
- 纯 AI 客服
- 陌生人/匿名聊天客服
- 只有网页嵌套的客服壳应用
- 只有一个输入框接机器人，没有实际业务页面的应用

## 7. 建议的首发 MVP 范围

以方案 A 为基线，首版建议控制在以下范围。

### 7.1 必做功能
- 手机号或邮箱登录
- FAQ 列表与搜索
- 提交工单
- 工单详情页
- 客服回复记录
- 图片上传（最多 3-6 张）
- 工单状态：待处理 / 处理中 / 已解决 / 已关闭
- 推送通知
- 设置页
- 隐私政策
- 用户协议
- 联系我们
- 账号注销

### 7.2 可延后功能
- 实时聊天
- 语音消息
- 多语言
- 深度 AI 自动应答
- 用户之间社区
- 评价体系
- 优惠券、营销中心

### 7.3 首版不要做
- 匿名聊天室
- 用户互相私聊
- 社区帖子流
- 虚拟道具/积分型聊天机制
- 没有业务归属的“在线陪聊式客服”

## 8. 页面规划

建议首版采用 4 个一级导航。

### Tab 1：首页
- 搜索框
- FAQ 分类
- 热门问题
- 发起工单 CTA
- 最近处理中的工单摘要

### Tab 2：工单
- 工单列表
- 状态筛选
- 工单详情
- 追加说明
- 查看客服回复

### Tab 3：消息
- 客服消息列表
- 系统通知
- 工单更新提醒

### Tab 4：我的
- 个人资料
- 联系我们
- 隐私政策
- 用户协议
- 通知设置
- 账号注销

如果选方案 B 或 D，可以把“工单”替换成“会话”或“订单”。

## 9. 交互与设计方向

结合当前 `DESIGN.md`，首版建议采用：
- 冷白背景 + 白色卡片
- 黑色标题 + 蓝灰色正文
- 大圆角 / 胶囊按钮
- 大留白、少色彩
- 首页突出“自助解决 + 人工支持”双入口

对客服类 App 来说，这种风格的好处是：
- 看起来更可信、更像正式工具
- 不会像社交聊天产品
- 更符合审核对“真实服务 App”的直觉预期

## 10. 技术实现建议

## 10.1 客户端
- Flutter 继续保留，优先覆盖 iOS
- 先从 `lib/main.dart` 拆出基础结构：路由、主题、页面目录、组件目录
- 首版先做本地假数据 UI，再接真实后端

### 10.2 后端方案建议：Node.js 主线

后端统一使用 **Node.js**，并与 Flutter 前端按 API 合同协作。首版建议采用“REST API + 数据库 + 对象存储 + 推送服务”的结构，后续如果做实时聊天，再加入 WebSocket。

#### 推荐技术栈
- Runtime：Node.js LTS
- Framework：NestJS 或 Express
- Database：PostgreSQL
- ORM：Prisma
- Auth：JWT + refresh token
- Storage：S3 兼容对象存储，用于工单图片
- Push：APNs / Firebase Cloud Messaging，用于 iOS 工单更新通知
- Realtime：Socket.IO 或原生 WebSocket，首版可延后

#### 首版后端模块
- 用户模块：注册、登录、刷新 token、退出登录、账号注销
- FAQ 模块：分类、问题列表、搜索、详情
- 工单模块：创建工单、上传附件、状态流转、追加说明、关闭工单
- 消息模块：客服回复、系统通知、未读数
- 文件模块：图片上传、访问控制、删除
- 合规模块：隐私政策、用户协议、数据删除记录
- 管理端接口：客服查看工单、回复工单、修改状态

#### 前后端配合方式
- Flutter 只负责展示、表单校验、登录态保存和请求 API
- Node.js 负责身份校验、数据权限、工单状态、内容存储和消息分发
- 前端先用 mock 数据完成 UI，再按接口文档替换为真实 API
- API 返回结构统一，避免每个页面单独处理错误格式

#### 推荐 API 结构
- `POST /auth/register`
- `POST /auth/login`
- `POST /auth/refresh`
- `POST /auth/logout`
- `DELETE /account`
- `GET /faqs`
- `GET /faqs/:id`
- `GET /tickets`
- `POST /tickets`
- `GET /tickets/:id`
- `POST /tickets/:id/messages`
- `POST /uploads/ticket-attachments`
- `GET /notifications`
- `PATCH /notifications/:id/read`

### 10.3 Node.js 后端方案对比

#### 方案 1：Express + Prisma（最快）
**优点**
- 简单直接，启动快
- 适合 MVP 和小团队
- 文件结构自由，开发成本低

**缺点**
- 后期模块变多时需要自己约束架构
- 权限、校验、异常格式需要自行规范

#### 方案 2：NestJS + Prisma（推荐）
**优点**
- 模块化清晰，适合客服系统长期迭代
- 依赖注入、守卫、管道、拦截器更适合做权限和统一响应
- 后续加 WebSocket、后台管理端、AI 服务更顺

**缺点**
- 初始代码量比 Express 多
- 学习和配置成本略高

#### 方案 3：Node.js + BaaS 混合
使用 Node.js 做业务 API，同时接 Firebase/Supabase 的 Auth、Storage 或 Push。

**优点**
- 减少自研基础设施
- 推送、存储、登录可以更快上线

**缺点**
- 数据和权限分散
- 隐私政策和数据流说明更复杂

### 10.4 我的建议
- 首版选 **NestJS + Prisma + PostgreSQL**，结构更适合客服应用长期维护
- 如果想极限提速，可以先用 **Express + Prisma**，但要提前规定目录、错误格式和权限中间件
- 实时聊天不要首版硬上，先用“工单消息 + 推送通知”完成客服闭环
- 前后端先共同确定 API 文档，再分别开发 Flutter 页面和 Node.js 接口

## 11. 上架合规清单

### 产品层
- [ ] App 中存在真实可完成的客服闭环
- [ ] 所有页面和按钮可用，没有空白占位
- [ ] 不做匿名/随机聊天
- [ ] 若有聊天，具备内容治理能力
- [ ] 客服联系方式明确可见

### 账号层
- [ ] 注册/登录流程稳定
- [ ] 退出登录可用
- [ ] 账号注销在 App 内可完成
- [ ] 游客模式是否允许要明确

### 隐私层
- [ ] 隐私政策可访问
- [ ] 数据收集目的清晰
- [ ] App Privacy 标签与实际一致
- [ ] 图片、通知等权限文案完整
- [ ] 删除数据流程明确

### 元数据层
- [ ] App 名称不夸张
- [ ] 截图为真实界面
- [ ] 描述不承诺无法保证的 SLA
- [ ] 年龄分级与聊天能力一致

### 技术层
- [ ] 崩溃和空数据场景处理完成
- [ ] 弱网下可用
- [ ] 推送可关闭
- [ ] iOS 权限申请只在使用时触发

## 12. 建议实施节奏

### 第 1 阶段：产品定型
- 确定你要服务的业务场景
- 选定方案 A / B / C / D
- 明确登录方式与数据字段
- 明确 FAQ、工单、消息的最小模型

### 第 2 阶段：iOS MVP
- 重构 Flutter 基础结构
- 完成首页、工单、消息、我的四大模块
- 接入登录、存储、上传、推送
- 接入隐私政策和账号注销

### 第 3 阶段：审核准备
- 完成 App Store 截图与文案
- 梳理 App Privacy 标签
- 准备测试账号
- 准备审核备注，说明这是品牌客户支持应用

### 第 4 阶段：上线后迭代
- 加会话实时刷新
- 加工单评价
- 加 AI 建议回复
- 加运营后台统计

## 13. 审核备注建议文案

提交审核时，建议在备注里用类似表述：

> This app is a customer support app for our own service. Users can search FAQs, submit support tickets, upload issue screenshots, and communicate with our support team regarding their own orders or service requests. The app does not provide anonymous or random social chat.

这类表述能主动把产品定位拉回“品牌客服工具”，降低误判概率。

## 14. 最终建议

如果你现在目标是：**尽快做出一个简单、真实、可上架的客服应用**，我建议你优先选：

1. **方案 A：FAQ + 工单 + 联系客服**，最稳、最快、最适合作为首发
2. 如果你本身有订单/售后体系，再升级为 **方案 D：订单驱动售后助手**
3. 不建议首发直接做开放式聊天或纯 AI 客服

## 15. 我建议你下一步直接确认的 5 件事

1. 这个 App 服务的是哪类业务：电商、SaaS、维修、预约、硬件，还是其他
2. 首版是否一定要实时聊天，还是先用工单
3. 是否已有后端系统可对接
4. 登录是手机号、邮箱，还是已有业务账号
5. 是否需要 AI 自动回复，还是先纯人工客服
