# Node.js 后端设计

## 1. 技术栈选择

首版建议采用：

- Runtime：Node.js LTS
- Framework：NestJS
- Database：PostgreSQL
- ORM：Prisma
- Auth：JWT access token + refresh token
- File Storage：S3 兼容对象存储
- Push：APNs / Firebase Cloud Messaging
- Validation：class-validator / Zod
- API Style：REST API
- Realtime：首版不做，后续再加 WebSocket 或 Socket.IO

如果你想更快起步，也可以用 Express + Prisma，但客服系统后续会自然扩展出权限、工单、消息、后台管理、推送、AI 等模块，所以更推荐 NestJS。

## 4. 角色与权限

### 4.1 用户角色

| 角色 | 说明 |
|------|------|
| USER | 普通 App 用户 |
| AGENT | 客服人员 |
| ADMIN | 管理员 |

### 4.2 权限原则

- USER 只能访问自己的工单、消息、通知和附件
- AGENT 可以访问被分配或待处理的工单
- ADMIN 可以访问管理端全部资源
- 所有写操作都必须校验身份
- 文件访问必须绑定资源权限，不能直接公开全部附件

| 错误码 | 说明 |
|--------|------|

## 9. 前端配合重点

Flutter 前端首版建议按以下方式对接：

- 登录成功后保存 `accessToken` 和 `refreshToken`
- 所有 API 请求自动附带 `Authorization`
- 收到 `UNAUTHORIZED` 时自动刷新 token
- 工单列表按状态筛选
- 工单详情使用消息流 UI，但底层仍是普通 REST 拉取
- 上传图片先调用上传凭证接口，再上传到对象存储，最后把 `attachmentId` 绑定到工单或消息
- 通知列表和未读数通过轮询或进入页面时刷新，首版不必做实时长连接

- 必须支持账号注销或数据删除流程
- 必须能说明聊天记录、图片、联系方式的存储目的
- 如果允许用户发文字或图片，后台要能处理举报和删除
- 附件 URL 不应永久公开，建议使用短期签名 URL
- 不要收集与客服无关的敏感数据
- 推送通知必须允许用户关闭
- 审核账号需要能看到完整客服流程

## 12. 推荐首版边界

首版后端只做：
- 账号
- FAQ
- 工单
- 工单消息
- 图片附件
- 通知
- 简单客服后台接口
- 举报与账号注销基础能力

首版不做：
- 实时聊天
- AI 自动回复
- 多客服排班
- SLA 自动升级
- 复杂权限系统
- 营销活动
- 用户社区

这样可以最大化上架稳定性，同时保留后续扩展空间。
